# 自律型AIディスカッション・プラットフォーム『roles』

## 1. システム概要

本システム『roles』は、ユーザーの抽象的な課題提起（例：「SIer営業の行動データ化」など）に対し、複数のAIエージェントが異なる役割（ロール）を担って仮想空間上で自律的に議論を行い、多角的な視点から要件整理や解決策を導き出す「Agentic Workflow」アプリケーションである。
ターミナルから `roles serve` コマンドを実行することでローカルサーバーが起動し、自動的にブラウザ上でリッチなWeb UIが展開される。

## 2. システム・アーキテクチャ

* **バックエンド兼APIサーバー:** Hono (Node.js または Bun環境を想定)
* **フロントエンド:** JSX + Tailwind CSS (CDN経由を許容)
* **LLMバックエンド:** OpenAI Compatible API (`/v1/chat/completions`)
  * 推論エンジンとして LM Studio, Ollama, または本家 OpenAI API をbase_url / api-key 切り替えのみでサポートできる。
* **通信方式 (UI更新):** SSE (Server-Sent Events) によるリアルタイム・ストリーミング描画。

## 3. UI/UX設計（3画面構成）

ユーザー体験を最大化するため、以下の3ペイン（ステップ）で画面を構成する。

1. **The Lobby（セットアップ画面）:**
   * ユーザーが議論したいテーマを入力するフォーム。
   * 入力後、AIが動的に選定した「参加者（ロール）」のカードがフェードインして表示される。
2. **The Arena（議論画面）:**
   * チャットUI形式。ファシリテーターや各ロールの発言が、SSEを通じてリアルタイムにタイピング（ストリーミング）表示される。
   * 画面脇に「現在の論点」と「裁定者（Judge）のステータス」をダッシュボード的に表示する。
3. **The Report（サマリー画面）:**
   * 議論終了後、議事録エージェントが生成した「決定事項」「対立意見」「残課題」がMarkdown形式の美しいレポートとして出力される。

## 4. ワークフロー定義（3フェーズ構成）

### フェーズ1：要求定義とロール生成（インタラクティブ・フェーズ）

1. ユーザーが初期テーマを入力。
2. **要件定義役**がユーザーと対話または入力内容から推定し、目的や制約条件を設定する
3. 情報が揃い次第、議論すべき「論点リスト」と「必要なロール（例: CxO, トップ営業, 現場のマネージャー・プレイングマネージャー, 情シス等）」をJSON形式で出力し、議論フェーズへ移行。

### フェーズ2：仮想会議の実行（オーケストレーション・フェーズ）

メインプロセス（Hono）が共通の会話履歴（Context）を保持し、以下のループを回す。

1. **ファシリテーター**が会話履歴を読み込み、次に発言すべきロール（または論点の提示）を決定・指名する。
2. 指名された**サブエージェント（ロール）**に対し、個別のSystem Promptと会話履歴をインプットしてAPIをコール。発言履歴を追加し、フロントエンドへSSEでストリーミング配信する。
3. 一定ターン経過後、**裁定者（Judge）**を呼び出し、論点が満たされたかをJSON（`{"is_resolved": boolean}`）で判定させる。
4. 未解決(`false`)なら手順1へ戻り、解決済み(`true`)ならループを終了してフェーズ3へ移行。

### フェーズ3：収束＆レポート出力（クロージング・フェーズ）

1. **議事録役**に対し、全会話履歴をインプットとして渡す。
2. 議論の要点、コンセンサス、人間が判断すべきオープンクエスチョンを抽出し、Markdownレポートとして出力・フロントエンドへ描画する。

## 5. コア・エージェント定義（System Prompt要件）

| エージェント名 | 役割と責務（System Promptの指示内容） | 出力形式 |
| :--- | :--- | :--- |
| **要件定義役** | ユーザーの曖昧な要求を構造化する。 | テキスト ＋ JSON |
| **ファシリテーター** | 議論を進行し、特定意見に偏らないよう発言権をコントロールする。 | テキスト |
| **サブエージェント** | （動的生成）与えられた立場に固執し、自部門の利益・視点から発言する。 | テキスト |
| **裁定者（Judge）** | 議論を俯瞰し、論点が解決されたかを客観的に評価する。 | JSON |
| **議事録役** | 長大な履歴からノイズを省き、要点と決定事項、残課題を抽出する。 | Markdown |

## 6. 非機能要件・制約事項（ガードレール）

1. **無限ループ防止機構（Circuit Breaker）:**
   * オーケストレーター（Hono側）で最大発言ターン数（例: 15ターン）をハードコードし、超過した場合は裁定者の判定を無視して強制終了・レポート出力へ移行する。
2. **コンテキスト長（トークン）の枯渇対策:**
   * 会話履歴が肥大化した場合、古い履歴を破棄するか、「これまでの要約」として圧縮して最新プロンプトに結合するメモリ管理を実装する。
3. **Structured Output（構造化出力）の徹底:**
   * 要件定義役や裁定者の出力は、アプリケーション側の条件分岐に使用するため、APIの `response_format: { "type": "json_object" }` などを活用し、必ずパース可能なJSONで取得する。
